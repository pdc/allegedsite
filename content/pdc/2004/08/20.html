Damian Cugley (20 July 2004)

MU part 3: More MUD

20 July 2004

Apologies to people reading this via LiveJournal’s syndicated feed; a combination of my software converting every header in to an RSS item and LiveJournal duplicating each item every time I edited the title has created a flurry of links to essays that I expect no-one but me has any interest in anyway.

Just to rub it in, here is some more of my imaginary generalized mark-up language MU and its metadata format MUD..

No-range tags

The property no-range tags is a list of etypes, as a YAML list: 

no-range tags:
  - img
  - embed
  - applet
  - br
  - hr
  - base
  - link

This is a kludge to allow for HTML’s empty elements: a list of etypes for which the no-range-tag syntax is permitted. For example, with the above definition, <hr> is treated the same as <hr/>.

MUDs vs. METAs vs. MUMs

In the previous section I said that MUDs include encoding and content-type information. Now I think it is important to split the MUD (information shareable amongst several documents) with document-specific information like character encoding. To this end I am proposing a simpler format, called META (for metadata). The format (which I shall not detail here) is essentially the intersection of YAML with RFC 822: header lines with a keyword followed by a colon and a value. The permitted keywords include:

  • type, the MIME media type, sans parameters;

  • charset, the character encoding scheme of the document this META file describes;

  • some others I cannot think of right now.

The point of the META is that it is very simple and it can be used to deduce the correct complete MIME content-type header. Web servers use META files attached to resources to tell them how to server said resources. On a Unix-like operating system, the convention is that for a given file foo/bar.baz, look for a corresponding file foo/bar.META, and failing that, look for baz.META (that is, use the file-name suffix to find a default META file).
