Suggestions for names in programs

Names used in programs should be consistent enough that one programmer can guess what the names used by another will probably be, and to reduce the mnemonic burden of keeping track of the many names. For example, it annoys me to have to remember to write 

WHERE DocRepMap.repId = Repositories.repositoryId

and not

WHERE DocRepMap.repId = Reps.repId

or the other possible combinations. I also find compound names like User.userName annoying—why not just have User.name?

Abbreviations

Programs are full of nouns used as class names and variable names.

For any given noun, there should be at most three spellings:

  • Full form: programme, module, repository, iterator, index
  • Short form: prog, mod, rep, iter, i
  • One or two letters: p, m, r, it, i

We can divide the categories of things being named according to longevity and how widely references are scattered throughout the programme. Exactly what goes in which category depends on the programming language in question. For example:

  • classes, global functions (these are used widely and last a long time)
  • member functions
  • member variables
  • local variables (these are short-lived)
  • loop counters (these are even more short-lived)

The rules are as follows:

  1. Things higher in this categorization should have longer (or equal length) names
  2. Things at the same level must use the same abbreviation for the same word

For example, if you have a member function programme, then the class name must be Programme not Prog, since class names are in a higher category. My preference is to use the 1-letter forms for loop variables, the abbreviated form for locals and for member variables, and the full forms for member functions and class names. For example: 

class Programme:
    ...
    def modules(self):
        if not self.mods:
            mods = []
            for m in db.allModules():
                if m.programme == self:
                    mods.append(m)
            self.mods = mods 
        return self.mods

(This is a lousy way to interrogate a database, but illustrates different spellings of the words module.)

In many cases the short form and the full name are identical. In some cases it makes sense for the 1-letter form and the short form to be the same. There is a school of thought that rejects all abbreviations: 

class Programme:
    ...
    def modules(self):
        if not self.cachedModules:
            modules = []
            for module_ in db.allModules():
                if module_.programme == self:
                    modules.append(module_)
            self.cachedModules = modules 
        return self.cachedModules

(The trailing underscore is required to avoid a clash with a keyword.) This still obeys the above rules, of course, and saves one from having to decide which spelling to use. On the other hand, I prefer to use shorter names as a hint that something is a temporary.

The second rule tells us that if we abbreviate expression as expr in one place, we should stick with that spelling, and not switch randomly between exp and expr. Note that XML abbreviates attribute as ATT in ATTLIST, thereby inconveniencing the hoards of programmers used to spelling it attr.

These rules apply within one domain. When combining libraries or protocols from different domains, we come in to positions where the same word means different things and is conventionally abbreviated differently. There’s not much we can do about this except to try to keep consitent within domains.

Collections

By collection I mean all the various data structures that contain lots of objects—lists, sets, bags, arrays, vectors, dictionaries, tables, etc. Collections should always have plural names:

mod = mods[i] # Python
Dim mod_ As Module_ = mods(i) ' VB .Net
Module mod = mods[i]; // C++ and its parodies

When I was a lad, functional programmers conventionally used x to an object of arbitrary type, xs for a list of whatever x was, xss for a list of xs, and so on. This makes sense when using the single-letter abbreviations of names. Thus 

ps = [m.Programme for m in mods] # list of programmes
tss = [m.Topics for m in mods] # list of list of topics
